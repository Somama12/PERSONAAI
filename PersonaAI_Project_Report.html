Abstract

PersonaAI is a full-stack web application designed to address a fundamental limitation of modern AI chat assistants: the inability to remember a user's personal context across sessions or reason over user-supplied documents. The core problem identified is that general-purpose AI tools treat every conversation as stateless, requiring users to repeatedly re-introduce themselves and their goals.

The application was implemented using a React (Vite) frontend and a Node.js/Express RESTful backend, with MongoDB as the persistent data store. The system integrates Google Gemini AI — specifically the gemini-2.5-flash model for conversational intelligence and the gemini-embedding-2-preview model for vector embeddings. Three core interaction modes were implemented: a Memory Mode in which AI-extracted long-term facts about the user are persisted and injected into every subsequent conversation, a Source Mode implementing a Retrieval-Augmented Generation (RAG) pipeline to allow users to query their own PDF documents and web URLs, and a General Chat Mode for open-ended conversation.

The main results demonstrate a fully functional system with persistent user authentication, autonomous memory extraction with AI-driven conflict detection, sentence-aware text chunking with cosine similarity search, and category-based conversation organization across six domains. In conclusion, PersonaAI successfully delivers a personalized AI assistant that grows more contextually aware the more a user interacts with it, establishing a strong foundation for future enhancements such as spaced-repetition learning and voice-based interaction.

Chapter 1: Introduction

The following chapter provides an overview of the project's motivation, the specific problem it addresses, the objectives established to solve it, and the scope of the implemented system.

1.1 Background and Motivation

Artificial Intelligence chat assistants have become ubiquitous in everyday academic and professional life. Tools such as ChatGPT, Google Gemini, and Claude are widely used by students to help them study, brainstorm ideas, and manage information. However, despite their impressive generative capabilities, all of these systems share a fundamental architectural constraint: they are stateless. Each conversation begins from scratch, with no recollection of prior sessions, user preferences, personal goals, or long-term context.

For a college student, this creates a frustrating experience. Each time a student returns to an AI assistant, they must re-introduce their project, their learning goals, their schedule constraints, and their personal background. Furthermore, when a student needs to extract insights from a specific textbook, a research paper, or a web article, there is no built-in mechanism to upload and query that document directly in a personalized context. The promise of AI as a truly personal intellectual partner remains unfulfilled.

1.2 Problem Statement

The core problem is the absence of persistent personal context in AI chat systems. When a user tells an AI "I am a biology student who struggles with chemistry," that information is forgotten the moment the session ends. Similarly, when a student uploads a PDF to summarize it, the results are generic rather than personalized to the student's background and goals. This forces repetitive re-introduction, reduces trust in the AI as a genuine assistant, and prevents the development of a personalized study or productivity workflow. Additionally, existing tools do not organize interactions by category (Work, Personal, Health, etc.), making it difficult to revisit and review past AI-assisted thinking on a specific topic.

1.3 Objectives

The primary goal of PersonaAI is to build a full-stack web application that acts as a personalized AI assistant — one that learns about the user over time, organizes conversations by category, and allows the user to query their own documents. The specific objectives are:

  • Persistent Memory: Automatically extract long-term personal facts from conversations and store them in a structured database. Inject these memories into every subsequent AI interaction so the assistant accumulates knowledge about the user over time.
  • Document Intelligence (RAG): Allow users to upload PDF files or provide web URLs. Process and embed these documents so users can ask natural-language questions answered strictly from the document's content.
  • Categorized Conversations: Automatically classify every conversation message into one of six domains — Work, Personal, Ideas, Learning, Health, or Miscellaneous — enabling organized retrieval and summarization.
  • Secure Multi-User Access: Support user registration and login with data isolation (each user's memories, messages, and sources are private).
  • Modern, Responsive UI: Deliver an intuitive, visually polished single-page application accessible from any device.

1.4 Scope and Limitations

PersonaAI targets college students and knowledge workers who want a personalized AI chat experience. The application is a web-based SPA (Single Page Application) backed by a REST API. It supports PDF and URL ingestion, three distinct chat modes, and persistent per-user memory.

Features intentionally outside the current scope include: OAuth/social login, native mobile applications, real-time collaborative features, voice input, LaTeX rendering for mathematical content, and advanced spaced-repetition learning algorithms. Additionally, password storage in the current implementation does not use hashing (a known limitation addressed in Future Work). The system also relies on external API quotas from Google Gemini, making very high-volume simultaneous use dependent on quota availability.

Chapter 2: Literature Review / Related Work

This chapter surveys existing AI assistant platforms and document querying tools, identifying the gaps that motivated the development of PersonaAI.

2.1 Existing Systems

ChatGPT (OpenAI)

ChatGPT, developed by OpenAI, is the most widely used AI assistant globally. It supports multi-turn conversations and, in its premium tier, offers limited memory features. However, the memory system is opaque — users cannot inspect, edit, or categorize stored facts. Free-tier users receive no memory at all. Furthermore, while ChatGPT supports file uploads, its document analysis is session-scoped and does not build a persistent, searchable knowledge base from the document.

Google Gemini (Google DeepMind)

Google Gemini offers strong multimodal capabilities and integrates with Google Workspace products. However, like ChatGPT, its memory features are limited and not user-controllable in a structured way. There is no mechanism to organize conversations by topic domain or to query a personal document library persistently.

Notion AI / Microsoft Copilot

Notion AI integrates AI into a note-taking workspace, providing context from the user's own notes. Microsoft Copilot connects AI to Microsoft 365 documents. Both systems are valuable but are tightly coupled to specific enterprise ecosystems, are expensive, and do not provide an open, programmable approach to personal memory or document Q&A.

LangChain / RAG Frameworks

Open-source frameworks like LangChain popularized the Retrieval-Augmented Generation (RAG) pattern, allowing developers to build document-querying systems. However, these are developer tools — not end-user applications. They have no built-in UI, no user accounts, and require significant engineering expertise to configure and deploy.

2.2 Gaps Addressed by PersonaAI

The analysis above identifies three critical gaps:

  • Transparent, User-Controlled Memory: No consumer AI tool allows users to see, edit, and delete the specific facts the AI has learned about them. PersonaAI's Memory Panel exposes every stored memory as an editable card.
  • Integrated Personal Document RAG: While RAG is well-established as a developer pattern, no accessible, free, user-friendly application bundles a complete RAG pipeline for personal documents. PersonaAI provides this behind a simple upload button.
  • Structured Conversation Organization: Existing tools treat all conversations as a flat, undifferentiated log. PersonaAI automatically categorizes every message and memory into six life domains, supporting organized review and per-category summarization.

Chapter 3: System Design / Methodology

This chapter describes the architectural design, technology choices, database schema, and user interface structure of PersonaAI.

3.1 System Architecture

PersonaAI follows a classic three-tier client-server architecture with a clear separation between the presentation layer (React SPA), the application/business logic layer (Node.js/Express REST API), and the data persistence layer (MongoDB). An additional cross-cutting concern — the Google Gemini AI service — is invoked exclusively from the backend, keeping API keys and AI logic out of the client.

Data MongoDB + Mongoose 9 Persistent storage for users, messages, memories, sources

The frontend communicates with the backend exclusively through HTTP REST calls using the browser's native fetch API. The backend validates all inputs, applies business rules, calls the Gemini API where required, and persists results to MongoDB. No AI processing happens on the client side.

The system is designed to be horizontally scalable: the backend can be deployed to any Node.js hosting platform (e.g., Render, Railway, AWS EC2), and the frontend can be hosted as a static bundle on Vercel, Netlify, or GitHub Pages. MongoDB Atlas provides a cloud-managed, horizontally shardable database backend.

3.2 Modules Description

A. Authentication Module

Manages user registration and login. The /api/signup endpoint validates that the email is unique before creating a new user document. The /api/login endpoint validates credentials and returns the user object. User session state is stored in the browser's localStorage, and the userId is sent with every subsequent API request to scope data appropriately.

B. Chat Module

Handles the core conversational AI experience. The /api/chat route accepts a user message and a mode flag. It retrieves the last 10 messages as conversation history and all stored memories as personal context. A single combined Gemini API call returns both the AI reply and the automatically detected message category in a structured JSON format — halving API quota consumption compared to making two separate calls.

C. Memory Module

Implements two-phase memory management. In the extraction phase (/api/memory/extract), the AI analyzes a piece of text and proposes a list of long-term personal facts. In the confirmation phase (/api/memory/confirm), the system runs an AI-powered conflict detection check against existing memories before saving. Users can view, edit, and delete memories via the Memory Panel in the dashboard.

D. Source Module

Implements the Retrieval-Augmented Generation (RAG) pipeline. Users can upload PDFs (/api/source/upload-pdf) or submit web URLs (/api/source/scrape-url). Both pathways extract text, apply sentence-aware chunking, generate vector embeddings in batches, and store everything in MongoDB. When a user queries a source (/api/source/query), the query is embedded and compared against stored chunk embeddings using cosine similarity, and the top-4 most relevant chunks are passed to Gemini to generate a grounded answer.

4.2 Backend and API Implementation

The backend is built with Node.js and Express 5. The server.js entry point initializes the Express application, connects to MongoDB via Mongoose, registers inline auth routes, and mounts the three modular route files: 

// server.js — route registration
app.use('/api/memory', memoryRoutes);
app.use('/api/source', sourceRoutes);
app.use('/api/chat', chatRoutes);

Each route module (e.g., chatRoutes.js) maps HTTP verbs and paths to controller functions. All data is exchanged as JSON. Input validation is performed at the controller level before any database or AI operations are invoked.

4.3 Memory Extraction Engine

The memory system uses a two-phase AI-powered workflow. In Phase 1, the extractMemory controller sends user text to Gemini with a structured prompt that asks it to return a JSON array of personal facts with a category and confidence score. Existing memories are included in the prompt to prevent duplicate extraction: 

// memoryController.js — extraction prompt (condensed)
const prompt = `
Extract core actionable facts or long-term personal details about the user.
If no important fact exists, output empty array [].
Otherwise, return JSON: [{ "fact": String, "category": String, "confidence": Number }]

Already known facts:
${memoryContext || "None"}

Text: "${text}"
`;
const response = await ai.models.generateContent({
  model: 'gemini-2.5-flash',
  contents: prompt,
  config: { responseMimeType: 'application/json' }
});

In Phase 2, confirmMemory runs an AI conflict-detection check before saving. If Gemini determines a new fact contradicts an existing memory, it returns the ID of the conflicting record and the system performs an update rather than an insert, keeping the memory store clean and non-redundant: 

// memoryController.js — conflict check
if (conflictData.conflict && conflictData.conflictingId) {
  const updated = await Memory.findByIdAndUpdate(
    conflictData.conflictingId,
    { content, confidence, updatedAt: new Date() },
    { new: true }
  );
  return res.status(200).json({ memory: updated, replaced: true });
}

4.4 RAG Pipeline (Source Mode)

The source module implements a complete Retrieval-Augmented Generation pipeline. When a PDF is uploaded, the raw binary buffer is parsed with pdf-parse. For URLs, the HTML is fetched and cleaned with Cheerio. In both cases, the resulting plain text is passed through a sentence-aware chunker: 

// sourceController.js — sentence-aware chunker
function chunkText(text, maxChars = 1200) {
  const normalized = text.replace(/\s+/g, ' ').trim();
  const sentences = normalized.match(
    /[^.!?]+[.!?]+["']?|[^.!?]+$/g
  ) || [normalized];
  const chunks = [];
  let current = '';
  for (const sentence of sentences) {
    if ((current + ' ' + sentence).length > maxChars && current.length > 0) {
      chunks.push(current.trim());
      current = sentence;
    } else {
      current = current ? current + ' ' + sentence : sentence;
    }
  }
  if (current.trim()) chunks.push(current.trim());
  return chunks;
}

Each chunk is passed to gemini-embedding-2-preview in batches of up to 100 to generate dense vector embeddings, which are stored alongside the chunk text in MongoDB. At query time, the user's question is embedded and cosine similarity is computed against every stored chunk: 

// sourceController.js — cosine similarity + top-k retrieval
function cosineSimilarity(vecA, vecB) {
  let dot = 0, normA = 0, normB = 0;
  for (let i = 0; i < vecA.length; i++) {
    dot += vecA[i] * vecB[i];
    normA += vecA[i] ** 2;
    normB += vecB[i] ** 2;
  }
  return dot / (Math.sqrt(normA) * Math.sqrt(normB));
}

// Top-4 most relevant chunks are retrieved and passed to Gemini
const scoredChunks = source.contentChunks
  .map(chunk => ({ text: chunk.text,
    score: cosineSimilarity(queryEmbedding, chunk.embedding) }))
  .sort((a, b) => b.score - a.score)
  .slice(0, 4);

4.5 AI Chat with Context Injection

The chat controller combines memory context and conversation history into a single efficient Gemini call. This design choice reduced API quota usage by 50% compared to the original two-call approach (one for categorization, one for the reply): 

// chatController.js — combined prompt with memory context
const combinedPrompt = `
You are PersonaAI, a helpful personal AI assistant with memory.
${memoryContext}

Conversation history:
${historyText}

Latest user message: "${text}"

Return ONLY a JSON object: { "category": "...", "reply": "..." }
`.trim();

const response = await ai.models.generateContent({
  model: 'gemini-2.5-flash',
  contents: combinedPrompt,
  config: { responseMimeType: 'application/json' }
});

Both the user message and the AI reply are persisted as Message documents in MongoDB with the automatically detected category, enabling future filtering and per-category summarization.

5.3 Discussion

The two-phase memory pipeline — extract then confirm — proved to be one of the most valuable design decisions. Rather than immediately saving every AI-proposed fact, the confirmation step gives the user visibility and control. The AI-powered conflict detection further prevents the memory store from accumulating contradictory or duplicate facts over time, which would degrade the context quality injected into future chats.

The single-call combined prompt (reply + categorize in one JSON response) was a significant optimization discovered mid-development. It halved Gemini API quota consumption while simplifying the server-side response parsing logic. The use of responseMimeType: 'application/json' in the Gemini config parameter constrains the model to produce valid JSON, eliminating the need for fragile regex-based extraction.

The sentence-aware chunker outperforms a naive fixed-size character splitter by preserving semantic units. Splitting at sentence boundaries means each embedded chunk represents a complete thought, improving cosine similarity accuracy and therefore the quality of retrieved context for RAG responses.

6.2 API Endpoint Validation

Backend endpoints were individually tested using direct HTTP requests to verify status codes, response shapes, and edge-case handling. Key validations included:

  • Missing required fields: All endpoints return 400 Bad Request with a descriptive message when required fields (e.g., userId, text) are absent.
  • Invalid userId: MongoDB lookups with non-existent user IDs return empty arrays gracefully rather than crashing.
  • Gemini API unavailable: When GEMINI_API_KEY is not set, all AI endpoints return 500 Internal Server Error with the message "AI not configured" rather than an unhandled exception.
  • PDF with no text layer: Scanned-image PDFs return empty chunks without crashing; the system saves a sourceless Source document.
  • Source not found: Querying a non-existent sourceId returns 404 Not Found.

6.3 Validation Strategy

Validation was performed at two levels. At the schema level, Mongoose enforces enum constraints and required fields, rejecting non-conforming documents before database write. At the business logic level, controllers validate request bodies before calling any AI or database operation. The AI response parsing is wrapped in try-catch with sensible fallback values (e.g., defaulting category to "Miscellaneous" if the AI returns an unrecognized value), ensuring the system degrades gracefully on model output variability.

Chapter 7: Conclusion and Future Work

7.1 Summary of Achievements

PersonaAI successfully achieved its primary objective: building a full-stack AI-powered personal assistant that maintains persistent memory across sessions and can intelligently query user-supplied documents. The system delivers a fully functional implementation of three distinct, novel capabilities rarely found together in a single open application:

  • Transparent AI Memory: An autonomous extraction, conflict-detection, and user-editable persistence pipeline that makes the AI genuinely context-aware across sessions.
  • Personal RAG System: A complete Retrieval-Augmented Generation pipeline over user-uploaded PDFs and web URLs, featuring sentence-aware chunking, batch vector embedding, and cosine similarity retrieval.
  • Organized Conversational Intelligence: Automatic six-domain categorization of all interactions with per-category summarization, turning an AI chat into a structured personal knowledge base.

The project also demonstrated effective API optimization, reducing Gemini API quota usage by 50% through a combined-call prompt design. The system was built with a clean separation of concerns — React SPA, Express REST API, MongoDB — making it straightforward to maintain, extend, or deploy.

7.2 Limitations

  • Password Security: The current implementation stores passwords in plain text. Industry-standard bcrypt hashing must be added before any production deployment.
  • No JWT Authentication: Session management relies on localStorage with a raw user object. Implementing JWT tokens would make sessions more secure and support token expiration.
  • Gemini API Quota Dependency: The system's availability is contingent on external API quota. Under heavy concurrent use, Gemini rate limits would throttle responses.
  • No Spaced Repetition: Memory retrieval injects all memories equally into context. A spaced-repetition algorithm could prioritize recently changed or low-confidence memories for more adaptive personalization.
  • Large Document Latency: Uploading large PDFs (100+ pages) involves extended batch embedding times. A background job queue (e.g., BullMQ) would improve the user experience by processing uploads asynchronously.
  • No Mobile App: The application is web-only; native iOS or Android apps with offline support are not available.

7.3 Future Work

  1. bcrypt Password Hashing + JWT: Migrate from plain-text storage to bcrypt-hashed passwords and issue signed JWT tokens with configurable expiration for all authenticated sessions.
  2. Spaced Repetition Memory Scheduling: Add an easeFactor and nextReviewDate to the Memory schema and implement an SM-2 scheduling algorithm to prioritize surfacing memories that are most relevant to the current context.
  3. Background Processing Queue: Introduce BullMQ or a similar job queue for PDF/URL ingestion to allow the upload API to return immediately and notify the user via WebSocket when processing completes.
  4. Voice Input: Integrate the Web Speech API for microphone-based message input, making PersonaAI more accessible and convenient on mobile devices.
  5. AI-Generated Flashcards: Allow users to generate study flashcards from any source document or conversation summary, bridging the gap between AI assistance and active recall learning.
  6. Shared Sources and Collaboration: Allow users to share a Source with other users — for example, sharing a course textbook within a study group — with privacy controls (public/shared/private).

Appendix B: Installation Guide

Prerequisites

  • Node.js v18+ and npm
  • A MongoDB Atlas account (or local MongoDB installation)
  • A Google Gemini API Key (from Google AI Studio)

Backend Setup

# 1. Navigate to the backend directory
cd PERSONAAI/backend

# 2. Install dependencies
npm install

# 3. Create a .env file in the backend directory:
#    MONGO_URI=mongodb+srv://<user>:<password>@cluster.mongodb.net/personaai
#    GEMINI_API_KEY=your_gemini_api_key_here
#    PORT=5001

# 4. Start the development server
npm run dev

Frontend Setup

# 1. Navigate to the frontend directory
cd PERSONAAI/frontend

# 2. Install dependencies
npm install

# 3. Start the Vite development server
npm run dev
